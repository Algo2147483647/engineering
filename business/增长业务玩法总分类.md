# 六、增长玩法的通用产品抽象

几乎所有增长玩法都可以抽象成下面这个结构：

| 模块 | 说明                                            |
| ---- | ----------------------------------------------- |
| 人群 | 谁能参与：新用户、会员、沉默用户、指定标签用户  |
| 行为 | 做什么动作：注册、浏览、分享、下单、支付、签到  |
| 条件 | 达成什么门槛：金额、次数、人数、时间、等级      |
| 奖励 | 给什么：券、积分、红包、会员、实物、权益        |
| 限制 | 限制什么：时间、库存、预算、次数、风控          |
| 触达 | 如何通知：Push、短信、弹窗、站内信、私域        |
| 核销 | 如何使用奖励：下单抵扣、兑换、提现、领取        |
| 追踪 | 如何衡量效果：曝光、点击、参与、转化、留存、ROI |
| 风控 | 如何防作弊：设备、手机号、IP、订单、支付、地址  |

所以，增长平台的底层模型可以是：

**人群规则 + 行为规则 + 奖励规则 + 限制规则 + 触达规则 + 风控规则 + 数据归因**

有了这个抽象，绝大多数玩法都不需要单独硬编码，只需要通过活动配置、规则引擎、权益系统和数据系统组合出来。
